# v0.4 Feature Batch Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Shadow-mode noop executor, pipeline run re-adoption after restarts (via store-persisted run handles), and a reference `.gitlab-ci.yml` for the central renovate-runner project.

**Architecture:** The Store interface gains a small run-handle KV (save/load/delete, both impls) used by the gitlabci executor to persist `{executor, platform, repo, project, pipelineID}` per active run; `gitlabci` implements `executor.Adoptable` by resuming pipeline polling from saved handles. The noop executor is a trivial `executor.Executor`. main reorders store construction before executors and wires both.

**Tech Stack:** existing only; no new dependencies.

## Global Constraints

- Same as v0.2 plan: Go 1.26, `-race` tests, `make lint` green per commit, conventional commits, no Co-Authored-By, executors honor ctx.
- Store handle semantics: one handle per repo key (per-repo lock guarantees a single active run). Redis handles carry the same TTL as repo entries; memory handles vanish on restart (status quo for memory users).
- Backwards compat: `gitlabci.New` signature changes (internal package, fine); config gains executor type `noop` only.

### Task 2: Store run handles + pipeline re-adoption

**Files:**
- Modify: `internal/store/store.go` (interface), `internal/store/memory.go`, `internal/store/redis.go`, `internal/store/semantics_test.go` (shared handle tests), `internal/store/redis_test.go` (TTL check)
- Modify: `internal/executor/gitlabci/gitlabci.go` + test
- Modify: `cmd/renovate-server/main.go` (store before executors; pass store to gitlabci.New)

**Interfaces:**

```go
// Store additions:
// SaveRunHandle persists executor-specific state for an active run.
SaveRunHandle(key, data string)
// LoadRunHandles returns all persisted handles (repo key -> data).
LoadRunHandles() map[string]string
// DeleteRunHandle removes a run's handle.
DeleteRunHandle(key string)

// gitlabci:
type HandleStore interface {
	SaveRunHandle(key, data string)
	LoadRunHandles() map[string]string
	DeleteRunHandle(key string)
}
func New(cfg config.Executor, client *gogitlab.Client, handles HandleStore, log *slog.Logger) (*Executor, error) // handles may be nil
func (e *Executor) AdoptRunning(ctx context.Context) ([]executor.AdoptedRun, error) // implements executor.Adoptable
```

Handle payload (JSON): `{"executor":"ci","platform":"gl","repo":"g/app","project":"infra/runner","pipelineID":42}`.

**Behavior:**
- `Run`: trigger pipeline → `SaveRunHandle(spec.Repo.Key(), json)` → poll (refactored into `pollPipeline(ctx, id)`) → `DeleteRunHandle` on any completion (deferred).
- `AdoptRunning`: for each handle whose `executor` matches `e.name`, return an `AdoptedRun` whose `Wait` polls the saved pipeline and deletes the handle when done. Corrupt handles are skipped with a warning and deleted.
- Redis: handles under `<prefix>handle:<repoKey>` with the configured TTL; memory: mutex map.
- main: construct store before executors; adoption loop (already present) now also picks up gitlabci.

**Tests:**
- store semantics suite: save→load→delete round-trip, load returns copy, overwrite updates.
- redis: handle key gets TTL (miniredis `mr.TTL`).
- gitlabci: (a) Run saves then deletes the handle (fake HandleStore records ops, assert payload fields); (b) AdoptRunning with a saved handle polls the existing pipeline to success WITHOUT hitting the trigger endpoint, then deletes the handle; (c) foreign-executor and corrupt handles ignored; (d) nil HandleStore keeps Run working.

**Steps:** failing store tests → store impls → failing gitlabci tests → gitlabci impl → main reorder/wiring → full tests + lint → commit `feat: re-adopt running pipelines after restart via store-persisted handles`.

## Self-Review Notes

- Handle lifecycle race: handle saved before polling starts; deleted in a defer covering every Run exit path including ctx timeout. Adopted waits delete in their own defer.
- Restart mid-trigger (handle saved, process dies before poll): handle persists → adopted next start. Restart before save: pipeline runs untracked → repo entry TTL heals (documented, unchanged).
- Memory-store users: LoadRunHandles empty after restart → adoption no-ops, exactly today's behavior.
- Dispatcher needs no changes: `Adopt` + `store.Adopt` already handle re-locking.
